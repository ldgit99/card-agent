# Harness 기반 수업설계 에이전트 구현 계획

기준 문서: [research.md](./research.md)
작성일: 2026-04-01  
현재 상태: 도메인 자료와 연구 문서는 준비되어 있고, 실제 앱 코드는 아직 시작 전이다.

## 1. 목표

이 프로젝트의 목표는 다음 두 가지를 동시에 구현하는 것이다.

1. 교사가 `설계 → 모의수업 실행 → 성찰` 흐름으로 수업을 설계하고 점검할 수 있는 웹 앱 구축
2. 그 앱을 안정적으로 발전시키기 위한 Harness 기반 개발 체계 도입

핵심 결과물은 다음이다.

- `Page 1`: 표 기반 수업 설계 화면
- `Page 2`: 멀티턴 모의수업 시뮬레이션 + 위험 탐지 + 성찰 일지
- `OpenAI Responses API` 기반 서버 백엔드
- `Vercel` 배포 가능 구조
- `Harness-Engineering`에서 선택 이식한 개발 하네스

## 2. 구현 원칙

- 연구 결과의 기준 문서는 항상 `research.md`로 둔다.
- 구현 시작 순서는 `데이터 모델 → seed 데이터 → Page 1 스캐폴딩 → 드래그앤드롭 → Page 2 → AI 연동`으로 고정한다.
- 브라우저는 UI와 상태 편집만 담당하고, OpenAI 호출은 서버 Route Handler에서만 처리한다.
- 핵심 결과 데이터는 모두 구조화한다.
- 카드 배치는 장식이 아니라 Page 2 시뮬레이션 입력 데이터다.
- MVP에서는 업로드 기능보다 정적 카드 seed 데이터와 설계 흐름 구현을 우선한다.

## 3. 작업 흐름 개요

전체 작업은 8개 단계로 진행한다.

1. 프로젝트 부트스트랩
2. 도메인 모델/seed 데이터 확정
3. Page 1 스캐폴딩
4. Page 1 드래그앤드롭 및 저장
5. Page 2 시뮬레이션 골격
6. OpenAI 연동
7. 성찰/저장/내보내기
8. Harness 도입 및 검증/배포

산출물:

## 5. 즉시 시작할 작업

다음 작업은 아래 순서로 바로 시작하는 것이 맞다.

1. `Next.js + TypeScript` 앱 스캐폴딩
2. `types/lesson.ts` 작성
3. `data/cards.ts` 작성
4. `Page 1` 표 기반 레이아웃 구현

이 순서가 가장 중요한 이유:

- 타입이 먼저 고정되어야 Page 1과 Page 2가 같은 데이터 구조를 쓴다.
- 카드 seed 데이터가 먼저 있어야 드래그앤드롭 UI를 붙일 수 있다.
- Page 2는 Page 1 출력 스키마가 확정된 뒤에 만드는 것이 비용이 적다.

## 7. 리스크와 대응

### 리스크 1. UI를 먼저 만들고 데이터 구조를 나중에 맞추는 경우

문제:

- Page 1과 Page 2 사이 계약이 흔들린다.

대응:

- `types/lesson.ts`를 먼저 작성하고, UI는 그 타입에만 맞춰 구현한다.

### 리스크 2. 카드가 장식 요소로 전락하는 경우

문제:

- 시뮬레이션이 카드 배치와 무관하게 동작한다.

대응:

- 카드 배치가 리스크 탐지와 성찰 질문 생성에 반드시 쓰이도록 강제한다.

### 리스크 3. OpenAI 호출 결과가 비정형으로 흐르는 경우

문제:

- UI 상태가 불안정해진다.

대응:

- Structured outputs를 강제하고, 파싱 실패 시 명시적 에러 처리한다.

### 리스크 4. 접근성 없는 드래그앤드롭 구현

문제:

- 교사 대상 도구임에도 품질 기준을 충족하지 못한다.

대응:

- `dnd-kit` 기반으로 키보드 센서, live region, row 연결 안내를 구현한다.

## 8. 수용 기준

- 20개 카드가 구조화 데이터로 존재한다.
- Page 1에서 `주제`, `교과`, `대상`, 활동 표 입력이 가능하다.
- 활동 row 선택 시 오른쪽 `인간 활동 카드`, `AI 카드` 박스가 해당 row와 연결된다.
- 카드 배치 결과가 상태 객체로 직렬화된다.
- Page 2는 Page 1 설계를 입력으로 읽는다.
- “AI가 답 제시, 교사는 결과 수용” 패턴에서 `AI 과의존`, `교사의 최종 판단 부재`가 탐지된다.
- 성찰 질문이 특정 턴과 연결된다.
- OpenAI 키가 브라우저에 노출되지 않는다.

## 9. 문서 운영 원칙

- 전략과 근거는 `research.md`를 기준으로 유지한다.
- 실행 순서와 현재 단계는 `plan.md`를 기준으로 유지한다.
- 구현이 시작되면 `progress.md`를 별도로 도입해 진행 상태를 추적한다.

## 10. 최종 권고

다음 시작점은 이미 결정되어 있다.

- `프로젝트 스캐폴딩`부터 한다.
- 곧바로 `types/lesson.ts`와 `data/cards.ts`를 만든다.
- 그 뒤 `Page 1`의 표 기반 입력 화면을 구현한다.

지금 단계에서 가장 피해야 할 것은 OpenAI 연동부터 붙이거나 Page 2를 먼저 만드는 것이다. 이 프로젝트는 Page 1의 입력 모델이 전체 시스템의 계약이므로, 첫 구현 스프린트는 반드시 **타입과 카드 데이터, 그리고 Page 1 구조 고정**에 집중해야 한다.
